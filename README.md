# Dialog Analyzer — система анализа текстовых диалогов

Проект предназначен для анализа клиентских текстовых диалогов. Система позволяет загружать CSV-файл с обращениями, автоматически определять тему диалога, эмоциональную окраску, находить проблемные сценарии и искать похожие обращения.

## Возможности

- Загрузка диалогов из CSV-файла.
- Классификация темы обращения.
- Эмоциональный анализ: позитивный, нейтральный, негативный.
- Выявление проблемных сценариев по совокупности признаков:
  - негативная эмоция клиента;
  - проблемная тема;
  - ключевые слова риска;
  - длина диалога;
  - количество вопросов;
  - признаки деэскалации.
- Расчет уровня критичности проблемы.
- Определение типа проблемы.
- Поиск похожих обращений через эмбеддинги и ChromaDB.
- Графики статистики по темам, эмоциям, критичности и типам проблем.
- Экспорт результатов анализа в CSV.

## Формат входного CSV

Файл должен быть в кодировке UTF-8 и содержать колонку `dialog`.

Пример:

```csv
dialog
"Клиент: Заказ не пришёл. Оператор: Проверим. Клиент: Спасибо."
"Клиент: Курьер опоздал, это недопустимо. Оператор: Извините."
```

## Запуск приложения

```bash
streamlit run app.py
```

После запуска откроется веб-интерфейс. В sidebar нужно загрузить CSV-файл с колонкой `dialog` и нажать кнопку **«Анализировать»**.

### Если используется proxy

Библиотека `httpx`, через которую HuggingFace загружает модели, не поддерживает proxy URL вида `socks4://...`. Приложение автоматически отключает такие переменные окружения на время запуска, чтобы не падать с ошибкой:

```text
ValueError: Unknown scheme for proxy URL URL('socks4://127.0.0.1:10808')
```

Если нужен proxy для загрузки моделей, используйте поддерживаемую схему, например `http://`, `https://` или настройте SOCKS5 с установленными дополнительными зависимостями среды.

## Результаты анализа

После обработки система добавляет к каждому диалогу поля:

| Поле | Описание |
|---|---|
| `тема` | Классифицированная тема обращения |
| `эмоция` | Эмоциональная окраска клиента |
| `проблемный` | Флаг проблемного сценария |
| `тип_проблемы` | Предполагаемый тип проблемы |
| `критичность` | Низкий, средний, высокий или критический уровень |
| `балл_проблемы` | Числовой риск-score |
| `причина` | Объяснение, почему обращение признано проблемным |

Результаты можно скачать кнопкой **«Скачать результаты CSV»**.

## Логика проблемных сценариев

Проблемность определяется не только по негативной эмоции. В `analyzer.py` используется скоринговая логика:

- негативная эмоция увеличивает риск;
- критичные ключевые слова дают высокий вклад в score;
- темы доставки, оплаты, возврата, качества и жалоб усиливают риск;
- много вопросов или длинный диалог добавляют дополнительный вес;
- слова вроде `спасибо`, `решено`, `помогли` уменьшают score как признаки деэскалации.

Уровни критичности:

| Score | Критичность |
|---|---|
| 0–1 | низкий |
| 2–3 | средний |
| 4–6 | высокий |
| 7+ | критический |

Диалог считается проблемным при score от 4.

## Поиск похожих обращений

Для поиска похожих диалогов используется модель:

```text
intfloat/multilingual-e5-small
```

Диалоги индексируются в ChromaDB. В интерфейсе можно ввести текст запроса, например:

```text
брак товара
задержка доставки
не вернули деньги
```

Система сначала ищет точные совпадения по ключевым словам, затем использует векторный поиск.

## Обучение моделей

В проекте уже есть готовые папки `topic_model/` и `emotion_model/`. Если нужно переобучить модели, сначала можно сгенерировать данные:

```bash
python generate_topic_data.py
python generate_emotion_data.py
```

Затем обучить модели:

```bash
python train_topic_model.py
python train_emotion_model.py
```

Модели сохраняются в папки:

```text
topic_model/
emotion_model/
```

## Текущие ограничения

- Обучающие данные в проекте синтетические, поэтому качество на реальных диалогах зависит от близости реальных формулировок к обучающим примерам.
- ChromaDB хранит данные локально в `chroma_data/`, поэтому при многократных загрузках возможны дубли в векторной базе.
- Для промышленного использования желательно добавить отдельную валидацию качества моделей, confusion matrix и тесты на реальных данных.
